# VSCode 内部版本（CodeArts Agent + 自研 AI 插件）黑屏问题 — 分析与修复

> 本文基于对 `gemini_context_1.md`、`gemini_context_2.md` 两份你与 Gemini 的完整聊天记录的提炼、交叉核对与再分析。
> 凡是“你实际采集到的证据（DMP / Perfetto / 启动参数 / code --status）”我都标为【已确定】；凡是 Gemini 推测、举例的签名我都标为【怀疑/推测】，不与你的真实日志混淆。
> 编写日期：2026-06-18

---

## 0. 一句话结论

**这是一个 Chromium/Electron 显示合成（Viz / Display Compositor）层面的“表面（Surface）唤醒同步失败”问题，被你们为绕开 GPU 兼容性问题而加的 `--disable-gpu-compositing`（强制软件合成）放大，并由“第二个互相通信的 Webview 改变了表面/遮挡拓扑”这一条件触发。** 进程没死、线程健康，只是合成管线在长时间后台/遮挡后醒来时拿到的是一张陈旧或空的聚合表面，于是整窗黑屏；`WM_SIZE`（改变窗口大小）或杀掉 GPU 进程会强制重建表面，因此能瞬间恢复 —— 这正是定位根因的“铁证”。

---

## 1. 问题总结

### 现象
- **整窗黑屏**：不是单个 Webview 区域发灰/发黑，而是**整个 VSCode IDE 主窗口变黑、画面冻结**（进程仍在、仍响应）。
- **触发条件**：把 VSCode **最小化 / 置于后台 / 被其它窗口遮挡较长时间（约 30 分钟级别）**后，再恢复/最大化到前台 → 黑屏。
- **关键前置**：**只有在插件加入“第二个 Webview”且两个 Webview 互相通信之后才出现整窗黑屏**。在只有一个 Webview 时，最多是单个 Webview 区域发黑，不会整窗黑。
- **恢复方式（两种，都极具诊断价值）**：
  1. **拖动 / 改变窗口大小**（触发 `WM_SIZE`）→ 立即恢复；
  2. **杀掉 GPU 进程** → 立即恢复。
- **平台**：Windows 10/11，x64。多显示器环境疑似更易复现。
- **不是崩溃**：DMP 显示线程都健康地停在消息泵等待，不是死锁、不是 crash。

### 涉及的自研改动
- 产品：`code.exe`（CodeArts Agent，内部定制 VSCode），Electron/Chromium **142**。
- AI 插件的 Webview 配置：`retainContextWhenHidden: true` + `enableScripts: true`。
- 插件行为：AI 在**后台**仍持续修改编辑器代码、并在两个 Webview 之间**高频 `postMessage` 通信**。

| C6 | Perfetto trace 中 `LayerTreeHostImpl::SetVisible` 标记为 **“(Did not end)”**（有始无终） | Perfetto proto trace | 强烈指向：可见性状态切换没走完，合成层认为自己“没被显示”。 |
| C7 | WinDbg 在 `RenderWidgetHostImpl::WasHidden` / `WasShown` 下断点可命中，Space 模式下各命中 **3 次**（workbench + 2 个 Webview） | WinDbg + Electron 142 PDB | 证明确实有 3 个渲染宿主参与可见性状态机；状态机路径可被监控。 |

| C10 | 加 `--no-sandbox` 测试，黑屏仍出现 | 实验 | **沙箱 / Win32k Lockdown 不是主因**（至少不是唯一开关）。 |
| C11 | “改变窗口大小恢复” 与 “杀 GPU 进程恢复” 均稳定有效 | 现场观察 | 见第 4 节根因推导，这是最关键的两条线索。 |

> 补充：即便加了 `--disable-gpu-compositing`，**GPU 进程依然存在**——因为 Chromium 的 Viz **显示合成器（Display Compositor）始终运行在 GPU 进程里**，软件合成只是把 `SoftwareOutputDevice` 放进 GPU 进程，最终聚合表面、推屏的仍是 Viz。这解释了为什么“杀 GPU 进程能恢复”。（见 [Chromium Viz 文档](https://chromium.googlesource.com/chromium/src/+/HEAD/components/viz/)）

## 4. 怀疑 / 假设（按可信度排序）与根因推导

### 主因（高可信）：软件合成模式下，长时间遮挡唤醒后的“表面同步失败”

把 C1/C2（软件合成）+ C6（SetVisible 未结束）+ C11（WM_SIZE 或杀 GPU 进程恢复）三条放在一起，指向同一条链路：

1. 窗口长时间最小化/被遮挡，Chromium 内部把渲染宿主标记为 `WasHidden`（C7 能命中），合成器 `SetVisible(false)`，**底层软件输出表面（`SoftwareOutputDevice` 的共享内存位图 / DIB）被释放或失效**（Windows DWM 在长时间后台后会回收窗口的图形缓冲）。
2. 恢复到前台时，本应走 `WasShown` → `Compositor::SetVisible(true)` → 重新分配 `LocalSurfaceId` → 子表面重新嵌入 → Viz 聚合新表面推屏。但在**软件合成 + 双 Webview（表面/遮挡拓扑更复杂）**的组合下，这条唤醒链**没走完**（C6 的 “SetVisible (Did not end)”）：
   - 要么父窗口的 `LocalSurfaceId` 没有重新分配/对齐，
   - 要么某个 Webview 子表面醒得慢、没被重新嵌入，
   - 要么 `SoftwareOutputDevice` 的输出位图没被重建。
3. 结果：Viz 显示合成器手里是一张**陈旧/空的聚合表面**，整窗推出黑色。但**进程、线程、DOM 全部健康**（C4），所以不是崩溃。
4. **为什么 `WM_SIZE` 能恢复**：尺寸变化强制重新分配 `LocalSurfaceId` 并触发整条 surface 重新嵌入 + 输出设备按新尺寸重建 → 管线被“踹”活。
5. **为什么杀 GPU 进程能恢复**：Viz 显示合成器整体在 GPU 进程里重建，重新和各 Renderer 建立 frame sink、重新聚合 → 推出新画面。

这与社区已知的一类 bug 行为一致：**occlusion 之后 LocalSurfaceId 处于 stale/未嵌入状态，可见性恢复时没有被重新激活**（参考 Chromium 官方 issue “Software compositing fallback for Viz” 40524580，以及树莓派上“右键遮挡后合成停摆、需重启才恢复”的同类现象）。

> 一句话：**“为了压下 GPU 兼容性这只葫芦，按出了软件合成器唤醒黑屏这只瓢。”** 关 GPU 合成本身就把你切到了 Chromium 里更冷门、更少被打磨的软件合成唤醒路径上。

### 协同放大因素（中可信）：后台高频 IPC + 双 Webview 改变了拓扑
- C9 证明纯 IPC/DOM 不能单独触发，但**双 Webview 互通**改变了 view/surface 树层级与唤醒时序，使 `WindowOcclusionTracker`/`SetVisible` 在恢复时更容易“误判仍遮挡 / bounds 瞬时为 0×0”。这解释了“为什么加了第二个 Webview 才从局部黑变成整窗黑”。
- 后台持续 `postMessage` 会让隐藏的 Renderer 背着一堆积压消息醒来，进一步挤占唤醒窗口期、加剧时序错位。

### 已基本排除
- **GDI 句柄耗尽** —— C5 排除。
- **进程崩溃 / 死锁** —— C4 排除。
- **沙箱 / Win32k Lockdown 为唯一主因** —— C10 部分排除。
- **纯 IPC / DOM 膨胀为唯一根因** —— C9 排除。

## 5. 修复方案（按“根治 → 兜底 → 缓解”分层，建议组合使用）

### 🥇 方案 A（根治，最高优先级）：不要用 `--disable-gpu-compositing`，换一条不致黑的合成路径
黑屏链路的起点就是软件合成。你们当初是为绕开“GPU 阶段兼容性问题”才禁的 GPU 合成 —— 但更好的做法是**保留 GPU 合成管线、只把底层 GL 换成软件实现或更稳的后端**，而不是把整条合成切到 GDI 软件路径。按优先级试：

1. **首选：用 SwiftShader 走软件 GL，但保留 GPU 合成管线**（避开软件合成器那条 bug 路径）：
   ```
   去掉： --disable-gpu-compositing
   改加： --use-gl=angle --use-angle=swiftshader
   ```
2. **次选：ANGLE + D3D11/WARP**（多数 Windows 兼容性问题用 WARP 即可，不必整条降级）：
   ```
   --use-angle=d3d11        # 或在 D3D 有问题时：--use-angle=d3d11-warp / --use-angle=d3d9
   ```
3. 如果当初的“GPU 兼容性问题”只是某些 feature，**精准用 blocklist/feature 关掉那一项**，而不是 `--disable-gpu-compositing` 一刀切。

> 验证目标：在恢复 GPU 合成（或 SwiftShader 合成）后，按原复现步骤（双 Webview + 后台 30 分钟 + 唤醒）能否还黑屏。若不黑 → 根因坐实在软件合成器唤醒路径，这条就是正解。

### 🥈 方案 B（兜底，几乎 100% 消除症状，强烈建议无论如何都加）：唤醒时强制重建表面
既然 `WM_SIZE` 能恢复，就在“窗口恢复可见 / 系统唤醒 / Webview 重新可见”时**主动制造一次极小的尺寸抖动**，强制 `LocalSurfaceId` 重分配与表面重嵌入。这是“曳光弹/急救包”，独立于根因都管用。

**Electron 主进程（最稳，作用于整窗）：**
```js
const { powerMonitor } = require('electron');

function nudgeResize(win) {
  if (!win || win.isDestroyed() || win.isMinimized()) return;
  const [w, h] = win.getSize();
  win.setSize(w + 1, h);   // 触发 WM_SIZE → 重新分配 LocalSurfaceId
  win.setSize(w, h);       // 立刻还原，用户基本无感
}

// 系统从睡眠/挂起唤醒
powerMonitor.on('resume', () => nudgeResize(mainWindow));
powerMonitor.on('unlock-screen', () => nudgeResize(mainWindow));

// 窗口从最小化/遮挡恢复、重新获得焦点
mainWindow.on('restore', () => nudgeResize(mainWindow));
mainWindow.on('focus',   () => nudgeResize(mainWindow));
mainWindow.on('show',    () => nudgeResize(mainWindow));
```

**插件侧（无法改 Electron 壳时的退路，作用于 Webview）：**
```js
// 在 Webview 的 HTML 脚本里
document.addEventListener('visibilitychange', () => {
  if (document.visibilityState === 'visible') {
    requestAnimationFrame(() => {
      // 强制一次 reflow / 重绘，踹活合成
      document.documentElement.style.transform = 'translateZ(0)';
      void document.documentElement.offsetHeight;            // 读取触发 reflow
      requestAnimationFrame(() => {
        document.documentElement.style.transform = '';
      });
    });
  }
});
```
```ts
// 扩展宿主侧：面板重新可见时主动“踹”一下
panel.onDidChangeViewState(e => {
  if (e.webviewPanel.visible) {
    e.webviewPanel.webview.postMessage({ type: '__forceRepaint' }); // Webview 收到后做上面的 reflow
  }
});
```

### 🥉 方案 C（消除“协同放大因素”，强烈建议）：后台不要再喂 IPC / 不要在隐藏态狂改 DOM
按可见性对 `postMessage` 做门控与缓冲，隐藏时只缓存、可见时批量补发：
```ts
class VisibilityGatedWebview {
  private buffer: any[] = [];
  private visible: boolean;
  constructor(private panel: vscode.WebviewPanel) {
    this.visible = panel.visible;
    panel.onDidChangeViewState(e => {
      this.visible = e.webviewPanel.visible;
      if (this.visible && this.buffer.length) {
        const batch = this.buffer.splice(0);            // 合并补发，避免唤醒瞬间洪泛
        this.panel.webview.postMessage({ type: 'batch', data: batch });
      }
    });
  }
  post(msg: any) {
    if (this.visible) this.panel.webview.postMessage(msg);
    else this.buffer.push(msg);                          // 后台只缓存不发
  }
}
```
- AI token 流式更新、两个 Webview 互通的消息，全部走这个门控。
- 后台修改编辑器代码可以继续（那是 Extension Host 的事），但**不要在隐藏的 Webview 上持续 append DOM**；同样用可见性门控。

### 方案 D（保底开关，验证用 / 应急用）：阻止渲染器进入“后台/遮挡”降级状态
作为 A 的对照实验，或在 A 落地前的临时缓解，加这组启动参数让渲染器别被挂起，从而绕开“遮挡→唤醒”那段坏掉的状态机：
```
--disable-backgrounding-occluded-windows
--disable-renderer-backgrounding
--disable-background-timer-throttling
```
> 代价：后台更费 CPU/内存。仅作过渡或 A/B 对照，不作为最终方案。

### 方案 E（最稳但有代价的退路）：`retainContextWhenHidden: false`
```ts
vscode.window.createWebviewPanel('id', 'AI', col, {
  enableScripts: true,
  retainContextWhenHidden: false,   // 隐藏即销毁 DOM，显示时重建
});
```
- 隐藏时销毁 Webview、显示时重建，**从机制上根除“隐藏态长期持有旧表面/旧 DOM”**，黑屏基本不可能再现。
- 代价：每次重新显示要重建 UI 状态（需把状态存到扩展宿主侧并回灌）。如果 A+B 已解决，可不动这个。

## 6. 落地与验证步骤（建议顺序）

1. **先做对照实验确认根因**：用方案 D 的参数（或方案 A 的 SwiftShader/ANGLE）跑一版，按复现步骤验证是否还黑屏。
   - 若“恢复 GPU/SwiftShader 合成后不黑、加回 `--disable-gpu-compositing` 又黑” → 根因坐实，**方案 A 作为长期解**。
2. **无论 A 结论如何，都加方案 B（唤醒 nudge）作为兜底** —— 成本极低、对用户无感、独立生效。
3. **加方案 C（可见性门控 IPC/DOM）** —— 消除放大因素，顺带降低后台资源占用。
4. 若 A 因业务原因无法去掉软件合成，则 **B + C + E** 组合，可在不依赖根因修复的前提下消除症状。
5. 回归测试：双 Webview + 后台 30 分钟 + 唤醒；多屏；最小化/Alt-Tab/锁屏唤醒 多场景各跑若干次。

## 7. 给团队的一句话同步

> 黑屏根因是 **Chromium 软件合成器（Viz Display Compositor）在长时间遮挡唤醒后的表面同步失败**，由我们为绕开 GPU 兼容性而加的 `--disable-gpu-compositing` 触发、被“第二个互通的 Webview”放大；进程不崩、线程健康，只是合成管线醒来时拿到陈旧/空表面，所以 `WM_SIZE`（拉伸窗口）或杀 GPU 进程能瞬间恢复。
> 修复：**(A) 不再强制软件合成，改用 SwiftShader/ANGLE 保留合成管线（根治）；(B) 窗口/Webview 恢复可见时强制 1px 尺寸抖动重建表面（兜底）；(C) 隐藏态对 postMessage/DOM 做可见性门控（去放大因素）。**
